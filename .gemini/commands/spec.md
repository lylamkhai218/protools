# /spec - PRD Definition Protocol

Use this slash-command equivalent to generate or evaluate a **Product Requirement Document (PRD)** before initiating structural additions.

## 🎯 Objectives
- Establish the absolute boundaries of the requested feature.
- Explicitly trace required inputs, expected outputs, user personas, and technical dependencies.
- Avoid scope creep by strictly defining **Non-Goals** (what is NOT to be implemented).

## 📄 PRD Template Structure

Every high-grade PRD must contain:
1. **Title & Descriptive Goal**: What are we building, and why?
2. **User Experience Flow**: Step-by-step interactive actions.
3. **Fidelity and UI Requirements**:
   - Typography pairings (e.g., Space Grotesk + JetBrains Mono).
   - Component structure and layout spacing boundaries.
4. **Data Schema & API Contracts**: Props interfaces, static assets, or state representations.
5. **Non-Goals / Structural Constraints**: Bulleted list of explicitly excluded attributes.
